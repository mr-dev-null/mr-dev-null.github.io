Unpacking a custom entropy packer in x64dbg

Contents

Warning: All analysis was conducted in an isolated, offline VM with no network adapters. Do not run malware samples on a connected or production machine.

Introduction

Recently I encountered a new variant of an info-stealer that used a custom packer to evade signature-based detection. The binary had a raw entropy of 7.8 — a strong indicator that the actual payload is compressed or encrypted inside a stub.

The goal of this post is to walk through the unpacking process step by step: from identifying the packer type, to locating the Original Entry Point (OEP), to dumping and rebuilding a clean import table.

Initial Recon

The first tool in the workflow is Detect It Easy (DIE). Throwing the binary in immediately tells us the packer is custom — no known signature matches. The entropy graph shows a single high-entropy section covering almost the entire file, which is consistent with a simple XOR or RC4 stub.

Info: Entropy above 7.2 on a PE section almost always means compressed or encrypted data. Entropy above 7.8 typically means encryption rather than compression.

Next, I look at the PE headers in PE-bear. The section names are scrambled — .packed instead of the usual .text / .data. The import table is stripped to almost nothing: only kernel32.dll with three imports — VirtualAlloc, VirtualProtect, and LoadLibraryA. This is the unpacking stub's minimal footprint.

Locating the OEP

The standard approach for this kind of packer is to set a hardware breakpoint on memory allocated by the stub, then wait for execution to transfer to the decrypted payload.

I set a breakpoint on VirtualAlloc and let it run. After the first call returns, I set a hardware execution breakpoint on the returned memory region. On the second VirtualAlloc call, the stub writes the decrypted payload and jumps to it. That jump target is the OEP.

x64dbg_bp.py 
import x64dbg

# Break on VirtualAlloc return, then set hardware BP on alloc'd region
def on_virtualalloc_ret():
    alloc_addr = x64dbg.reg.get_val("rax")
    size       = x64dbg.reg.get_val("rdx")

    if alloc_addr:
        x64dbg.dbg.set_hardware_breakpoint(
            alloc_addr, 1,  # 1 = execute
        )
        print(f"[*] Hardware BP set at {hex(alloc_addr)}")

    return True

Running with this script attached, execution breaks inside the newly allocated region. A quick look at the disassembly confirms we're at a valid PE entry point — the classic sub rsp, 28h prologue is visible.

Dumping & Rebuilding

With execution paused at the OEP, I use Scylla (integrated in x64dbg) to dump the process. The import table at this point is fully resolved in memory — Scylla can walk the IAT and rebuild it automatically.

After dumping, I run the result through DIE again. Entropy is now 5.9 — a normal value for an unpacked PE. The import table shows the full set of Win32 API calls, confirming the dump is clean.

The packer was straightforward, but the technique generalizes well. Any packer that relies on VirtualAlloc + VirtualProtect to decompress a payload can be approached the same way.

Conclusion

Custom packers that minimize their import footprint to VirtualAlloc / VirtualProtect / LoadLibraryA are one of the most common evasion patterns in current malware families. The hardware breakpoint approach is reliable and doesn't require any packer-specific knowledge upfront.

The unpacked binary turned out to be a variant of a credential harvester targeting browser credential stores. I'll cover the inner workings of the payload in a follow-up post.
